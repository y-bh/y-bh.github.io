JMM内存模型

JMM（Java Memory Model）本身为一种抽象概念并不真实存在，其描述的是一组规则或规范，通过这组规范定义了程序中各个变量(包括实例字段，静态字段和构成数组对象的元素)的访问方式

JVM运行程序的实体为线程，每个线程创建时JVM都会为其创建一个工作内存（逻辑上），为线程私有。

Java内存模型规定所有变量都存储在主内存，主内存是共享内存区域，所有线程都可访问。

线程对变量的操作步骤为：

  1. 从主内存中读取变量值到工作内存
  2. 对拷贝的值进行操作
  3. 将值写回给主内存

因为不能直接对主内存的变量值进行修改，因此各个线程中存着主内存变量的拷贝副本。但这样不同线程间的通信也要通过主内存来实现。

JMM模型需要保证：

  • 可见性
  • 原子性
  • 有序性

原子性

原子表示不可分割，因此原子性为不被中断操作。即，当一个线程开始操作后，不会被其他线程干扰导致中断。

原子性问题一般为：当线程1、2同时取到了主内存中的值，并在各自的工作内存中进行修改，当要对主内存中的变量进行修改时，线程1的时间片结束。此时线程2修改了主内存的值。而后线程1恢复正常，将修改的值再次写入主内存，造成主内存的值写入了两次，但是值的内容只修改了一次的现象。

可见性

当JMM中其中一个线程修改了主内存中变量的值的时候，其他线程需要能够立刻得知修改的值。

可见性问题一般为：当线程1修改了变量值想要刷新到主内存中时，线程2读取了主内存中的值，此时线程2读取的是修改前的，读取出错。

有序性

线程内操作有序进行，线程间操作有序进行。

有序性问题一般为：有时写的Java代码的执行顺序并不是按照写的顺序来的。编译器会为了执行效率进行重排序。处理器也可能为了提升性能，对编译器生成的指令进行重排序。

重排序原则

as-if-serial语义：重排序之后的代码在单线程环境下执行的结果不能改变。

JMM提供了三种方式来保证有序性：

  • happens-before原则
  • synchronized机制
  • volatile机制

synchronized有序性原理

synchronized保证有序性的原理很好理解。因为synchronized可以保证同一时间只有线程能访问代码块，而在单线程环境下，JMM能天然保证代码的串行语义。

happens-before原则

happens-before原则是Java内存模型中定义的两项操作之间的偏序关系，如果说操作A先行发生于操作B，其实就是说在发生操作B之前，操作A产生的影响能被操作B观察到。“影响”包括修改了内存中共享变量的值、 发送了消息、 调用了方法等。

Java内存模型存在一些先行发生关系，这些先行发生关系无须任何同步器协助就已经存在，可以在编码中直接使用。 如果两个操作之间的关系不在此列，并且无法从下列规则推导出来的话，它们就没有顺序性保障，虚拟机可以对它们随意地进行重排序

  1. 程序次序规则（Program Order Rule）：在一个线程内，按照程序代码顺序，书写在前面的操作先行发生于书写在后面的操作。 准确地说，应该是控制流顺序而不是程序代码顺序，因为要考虑分支、 循环等结构。
  2. 管程锁定规则（Monitor Lock Rule）：一个unlock操作先行发生于后面对同一个锁的lock操作。 这里必须强调的是同一个锁，而“后面”是指时间上的先后顺序。
  3. volatile变量规则（Volatile Variable Rule）：对一个volatile变量的写操作先行发生于后面对这个变量的读操作，这里的“后面”同样是指时间上的先后顺序。
  4. 线程启动规则（Thread Start Rule）：Thread对象的start()方法先行发生于此线程的每一个动作。
  5. 线程终止规则（Thread Termination Rule）：线程中的所有操作都先行发生于对此线程的终止检测，我们可以通过Thread.join()方法结束、 Thread.isAlive()的返回值等手段检测到线程已经终止执行。
  6. 线程中断规则（Thread Interruption Rule）：对线程interrupt()方法的调用先行发生于被中断线程的代码检测到中断事件的发生，可以通过Thread.interrupted()方法检测到是否有中断发生。
  7. 对象终结规则（Finalizer Rule）：一个对象的初始化完成（构造函数执行结束）先行发生于它的finalize()方法的开始。
  8. 传递性（Transitivity）：如果操作A先行发生于操作B，操作B先行发生于操作C，那就可以得出操作A先行发生于操作C的结论。

时间先后顺序与先行发生原则之间基本没有太大的关系，所以衡量并发安全问题的时候不要受到时间顺序的干扰，一切必须以先行发生原则为准。

Volatile

Volatile是Java虚拟机提供的轻量级的同步机制，其特点为：

  • 保证可见性
  • 不保证原子性
  • 禁止指令重排序

volatile保证可见性原理

缓存

每条指令都是在CPU中执行的，其过程势必涉及到读写数据，而程序运行的数据写在主内存中，这样CPU的执行指令的速度要高于读写主内存中的数据。因此，现代处理器为了提高内存数据的访问速度，会自带多级缓存，其位置在内存与处理器之间。

老的CPU会有两级缓存（L1和L2），新的CPU会有三级缓存（L1，L2，L3 ）。L1的缓存分为两种，一种为指令缓存，一种是数据缓存。L2、L3则不分缓存和数据。

L1、L2缓存在每一个CPU核中都有一份，L3则是所有CPU核心共享的内存。

离CPU越近速度越快，但内存越小。

速度上：L1>L2>L3>RAM内存。大小上相反。

可见性问题

虽然缓存加快了处理器访问内存的速度。但每个 CPU 缓存中有一份自己的内存副本，会带来各个 CPU 在访问同一块内存的数据时，每个 CPU 缓存中的副本可能不一致。

解决方案

使用MESI(缓存一致性协议)来解决可见性问题。

MESI协议将缓存数据分为四个状态：

  • Modified(已修改)：变量改变
  • Exclusive(独占的)：只有一个CPU有此变量
  • Shared(共享的)：变量同步到所有Cache中
  • Invalid(无效的)：当其中一个CPU中的值变为Modified。其余CPU中的值状态变为Invalid

MESI 这种协议在数据更新后，会标记其它共享的CPU缓存的数据拷贝为Invalid状态，然后当其它CPU再次read的时候，就会出现 cache miss 的问题，此时再从内存中更新数据。

总结为：其中一个线程更新成功→其余CPU缓存中的值标为Invalid(无效)→其余CPU读取缓存miss→去主内存中获取新值。

优化

从主内存中获取新值相比于缓存速度会慢很多，可以考虑从修改主内存的CPU缓存中读取数据。但控制状态也会变得麻烦，需要新加一个状态：Owner(宿主)。主要用于标记更新主内存的源CPU。

此方式，即为MOESI协议。因为允许 CPU Cache 间同步数据，所以降低了对内存的操作，性能是非常大的提升。

volatile保证有序性原理

处理器除了缓存外，计算单元与缓存系统之间还包括：本地寄存器、缓冲区。

若处理器本身完成了计算但是没将新值刷新到缓存中。缓存一致性协议就失去了作用。数据的新值对其他处理器的可见性依旧无法得到保证。

因此，要解决的是将计算单元计算的新值及时刷新到缓存中，这是单个处理器要解决的问题，依赖于内存屏障机制。

内存屏障机制

有两个功能：

  • 阻止屏障前后的没有数据依赖性的代码进行指令重排序，保证程序在一定程度上的有序性。
  • 强制把写缓冲区/高速缓存中的脏数据等写回主内存，让缓存中相应的数据失效，保证数据的可见性。

写内存屏障(Store Memory Barrier)：强制所有在store屏障指令之前的store指令，都在该store屏障指令执行之前被执行，并把store缓冲区的数据都刷到CPU缓存。这会使得程序状态对其它CPU可见，这样其它CPU可以根据需要介入。

读内存屏障(Load Memory Barrier)：强制所有在load屏障指令之后的load指令，都在该load屏障指令执行之后被执行，并且一直等到load缓冲区被该CPU读完才能执行之后的load指令。这使得从其它CPU暴露出来的程序状态对该CPU可见

总结：

依照volatile机制：

当第二个操作是volatile写时，不管第一个操作是什么，都不能重排序。这个规则确保volatile写之前的操作不会被编译器重排序到volatile写之后。
当第一个操作是volatile读时，不管第二个操作是什么，都不能重排序。这个规则确保volatile读之后的操作不会被编译器重排序到volatile读之前。
当第一个操作是volatile写，第二个操作是volatile读时，不能重排序。

因此：

  • 写volatile变量时，可以确保volatile写之前的操作不会被编译器重排序到volatile写之后。
  • 读volatile变量时，可以确保volatile读之后的操作不会被编译器重排序到volatile读之前。

AtomicInteger

原子类，保证代码的原子性。

多数时候，一个递增或递减的方案需要满足简单、高效、线程安全的需求。虽然synchronized或Lock可以满足功能上要求，但是对于代码上显得冗余，并且影响性能。

CAS(compare and swap)

比较并交换，是原子操作的一种，可用于在多线程环境下实现不被打断的数据交换操作，从而避免多个线程同时修改同一数据时，由于执行顺序的不确定性以及中断的不可预知性产生的数据不一致问题。

CAS中包含三个操作数：

  • 内存值V
  • 预期值A
  • 待更新值B

当预期的值A和内存中的值V相同时，将V修改为B。

当使用CAS在多线程场景下更新同一变量时，只有其中一个线程能更新变量的值。其他失败的线程并不会挂起，可以再次尝试获取(或什么也不做)

实现原理

CAS体现在sun.misc.Unsafe类的各个方法中。调用Unsafe的CAS方法，JVM会实现CAS汇编指令。

Unsafe中的CAS实现过程：

其中的compareAndSwapInt()是native方法，借助非Java语言的方式来实现。主要原因在于JAVA无法对操作系统底层进行操作，但是可以通过jni(java native interface)调用其他语言来实现底层的访问。

而CAS是一种系统原语(原语属于操作系统用语范畴，由若干条指令组成的，用于实现某个功能的一个过程)。原语的执行必须是连续的，在执行过程中不允许被中断。因此，不会造成数据不一致问题。

缺点

  • 若线程修改CAS值失败，便会一直尝试获取，如果长时间不成功，会大量增加CPU的开销。
  • Java中的CAS操作只是对CPU的cmpxchgq指令的一层封装，功能就是一次只原子地修改一个变量。当对多个共享变量进行操作时，CAS无法保证操作的原子性，此时可用锁来实现。
  • ABA问题

ABA问题

  1. 当两个线程A、B同时获取主内存中的值100，并在各自的缓存中生成了内存副本。
  2. 此时A挂起，B通过CAS将主内存的值更新为200，并又进行了第二次更新，将主内存的值更新为100。
  3. A恢复，继续执行先前的更新操作，自身的预期值为100，而主内存的值恰好为100，这时A认为主内存中的值没被更改过，进而进行更新操作。

虽然操作成功，但过程出现了问题。

例如：当使用场景变为链表，线程A、B都需要删除链表中的某一节点。当A挂起，B删除节点后又新加了一个节点，该节点恰好插入原删除的节点时，A恢复，执行删除操作时，因为两个节点的地址是相同的，所以操作会执行成功，将新增加的节点删除。

ABA解决方案

通过使用AtomicStampedReference类和AtomicMarkableReference类来解决。

AtomicStampedReference类

通过增加一个时间戳来进行版本的计数，可以得知引用变量中途改变了几次。

AtomicMarkableReference类

和AtomicStampedReference类相比，不再用Int标识引用，而是使用Boolean变量表示是否更改。

总结

二者都可解决ABA问题，根据计数和Boolean的特性区分场合使用。

使用该类后，同一场景下，若B更新主内存的数据后，时间戳会增加。当A恢复进行操作时，A的时间戳还是从主内存取出时的状态，和主内存的不一致，因此操作失败。

集合类线程安全

当使用线程不安全的集合类时，会有并发修改异常的问题。

异常：java.util.ConcurrentModificationException

因为ArrayList、LinkedList并不是线程安全的。所以，实现此功能需要保证集合的线程安全。

具体的解决方式为：

  • 使用Vector类，该方法内部使用synchronized加锁
  • 使用工具类Collentions.synchronizedList(new ArrayList<>());的方法创建集合。
  • 使用CopyOnWriteArrayList实现

CopyOnWrite(写时复制)

Copy-On-Write简称COW，是一种用于程序设计中的优化策略。

所有线程共享同一变量，当某一线程想对共享变量进行操作时，会将变量复制一份，在复制的变量中进行操作。操作结束后，将原来的变量的引用指向新的变量。

如此一来，当线程进行读操作时，因为不对变量进行修改操作，因此不需要加锁，支持并发的读。当进行写操作时，遵循COW的规则来防止线程安全问题。

JDK5往后，提供了CopyOnWriteArrayList和CopyOnWriteArraySet来实现COW容器。

CopyOnWriteArrayList

使用写时复制技术实现线程安全的ArrayList类。

Add方法中，先复制出新的数组，再将新元素加到新数组中，再将原来的数组的引用指向了新数组，符合COW的策略。

remove方法中，需要判断删除的元素是否为最后一个。如果是，直接在复制的时候将数组长度-1即可。

CopyOnWriteArraySet

即，线程安全的HashSet。但是，虽然该类和HashSet都继承于AbstractSet类。但是：

  • HashSet底层通过HashMap(散列表)来实现
  • CopyOnWriteArraySet底层通过CopyOnWriteArrayList实现

CopyOnWriteArraySet的结构严格意义来说是一个集合，它的底层实现是利用数组，它的上层实现是CopyOnWriteArrayList。

CopyOnWriteArraySet是一个集合，所以它是不可以放置重复的元素的，它的取重逻辑是在add中体现的。

CopyOnWriteArraySet是利用CopyOnWriteArrayList来实现的，因为CopyOnWriteArrayList是线程安全的，所以CopyOnWriteArraySet操作也是线程安全的。

ConCurrentHashMap

ConcurrentHashMap(JDK7)

底层结构

Segment数组+HashEntry数组+链表(HashEntry节点)

Segment数组中存储Segment对象，每个Segment中存储一个或多个HashEntry数组，数组中存储每个HashEntry链表的头节点

还有两个主要的内部类：

  • class Segment内部类，继承ReetranLock。

ConcurrentHashMap 的扩容是仅仅和每个Segment元素中HashEntry数组的长度有关，但需要扩容时，只扩容当前Segment中HashEntry数组即可。也就是说ConcurrentHashMap中Segment[]数组的长度是在初始化的时候就确定了，后面扩容不会改变这个长度。

默认并发级别会创建包含 16 个 Segment 对象的数组。每个 Segment 的成员对象 table 包含若干个散列表的桶。每个桶是由 HashEntry 链接起来的一个链表。如果键能均匀散列，每个 Segment 大约守护整个散列表中桶总数的 1/16。

  • class HashEntry内部类

用来封装散列映射为表中的键值对，key，hash和 next 域都被声明为 final 型，value 域被声明为 volatile 型

在 ConcurrentHashMap 中，在散列时如果产生“碰撞”，将采用“分离链接法”来处理“碰撞”：把“碰撞”的 HashEntry 对象链接成一个链表。由于 HashEntry 的 next 域为 final 型，所以新节点只能在链表的表头处插入

主要方法

get()

  1. 找到对应的Segment，使用Unsafe类的getObjectVolatile方法，具有读的volatile语义，保证多线程下线程安全。
  2. 获取hashentry[],由于table是每个segment内部的成员变量,使用volatile修饰的，所以也可以保证一致性。
  3. 获取具体的hashentry,也使用了UNSAFE的getObjectVolatile具有读的volatile语义,然后遍历查找返回.

put()

  1. 根据key计算出hashcode
  2. 确定segment数组的位置：hashcode & segments.length-1
  3. 确定HashEntry数组的位置： hashcode & HashEntry.length-1

执行put方法插入数据时，根据key的hash值，在Segment数组中找到相应的位置，如果相应位置的Segment还未初始化，则通过CAS进行赋值，接着执行Segment对象的put方法通过加锁机制插入数据。

总结

利用Unsafe操作+ReentrantLock+分段思想，实现线程安全。

Unsafe中使用了：

  • compareAndSwapObject：通过CAS的方式修改对象的属性
  • putOrderedObject：并发安全的给数组的某个位置赋值
  • getObjectVolatile：并发安全的获取数组某个位置的元素

当调用ConcurrentHashMap的put方法时，最终会调用到Segment的put方法，而Segment继承了ReentrantLock，所以Segment自带可重入锁。它会先利用可重入锁加锁，成功后进行插入操作，完成后解锁。

底层实现过程：

ConcurrentHashMap底层是由两层嵌套数组来实现的:

  1. ConcurrentHashMap对象中有一个属性segments，类型为Segment[]
  2. Segment对象中有一个属性table，类型为HashEntry[]

当调用ConcurrentHashMap的put方法时，先根据key计算出对应的Segment[]的数组下标j，确定好当前key,value应该插入到哪个Segment对象中，如果segments[j]为空，则利用自旋锁的方式在j位置生成一个Segment对象。然后调用Segment对象的put方法。
Segment对象的put方法会先加锁，然后也根据key计算出对应的HashEntry[]的数组下标i，然后将key,value封装为HashEntry对象放入该位置，此过程和JDK7的HashMap的put方法一样，然后解锁。

ConcurrentHashMap(JDK8)

底层结构

Node数组+链表/红黑树

Node数组用来存放树或者链表的头结点，当一个链表中的数量到达一个数目时，会使查询速率降低，所以到达一定阈值时，会将一个链表转换为一个红黑二叉树，加快查询的速率

主要属性

jdk1.8中虽然不在使用分段锁，但是仍然有Segment这个类，但是没有实际作用

内部类结构

Node：
ConcurrentHashMap存储结构的基本单元，实现了Map.Entry接口，用于存储数据。Node的val和next都用volatile修饰，保证可见性。它不允许调用setValue方法直接改变Node的value域，它增加了find方法辅助map.get()方法。

TreeNode：
继承于Node，但是数据结构换成了二叉树结构，它是红黑树的数据的存储结构，用于红黑树中存储数据，当链表的节点数大于8时会转换成红黑树的结构，他就是通过TreeNode作为存储结构代替Node来转换成红黑树

如何保证线程安全

使用的是优化的synchronized 关键字同步代码块 和 cas操作了维护并发。

为什么要用synchronized，cas不是已经可以保证操作的线程安全吗？

CAS也是适用一些场合的，比如资源竞争小时，是非常适用的，不用进行内核态和用户态之间的线程上下文切换，同时自旋概率也会大大减少，提升性能，但资源竞争激烈时（比如大量线程对同一资源进行写和读操作）并不适用，自旋概率会大大增加，从而浪费CPU资源，降低性能。

锁

公平锁&非公平锁

公平锁：多个线程按照申请锁的顺序来获取锁

非公平锁：不按照申请的顺序获取锁，

在ReenTrantLock中，默认为非公平锁，可以设置为公平锁。

在Synchronized中，只能为非公平锁。

非公平锁实现

非公平锁只进行了CAS的判断，保证原子性。对于线程的竞争并没有进行约束。

公平锁实现

公平锁中加入了hasQueuedPredecessors()的判断，该方法作用为：

  • 当前线程之前有等待队列线程则判断为true。
  • 当前线程在队列的头部或队列为空则判断为false

当该条件通过后才能获取到锁。因此，只有队列中最前面的线程才能获取到锁资源。

总结

  1. 若释放锁的时候，没有新的线程来获取锁，则非公平锁等于公平锁
  2. 若释放锁的时候，正好有新的线程来获取锁资源。此时位于队列头的线程还没有被唤醒(这是可能的，因为线程上下文切换是需要不少开销的)，此时后来的线程优先获得锁，打破了公平。

其实，只要是线程进入了等待队列，队列里的线程依旧遵循FIFO原则，和公平锁的顺序相同。因为公平锁与非公平锁的release()部分代码，是共用AQS的。

非公平锁效率高于公平锁的原因在于非公平锁减少了线程挂起的几率，后来的线程有一定几率避免被挂起的开销。

可重入锁(递归锁)

java的锁都是可重入锁。

指以线程为单位，当一个线程获取对象锁后，这个线程可以再次获取本对象上的锁，而其他的线程是不可以的。

其意义在于，可以防止死锁。例如，当子类对象使用super关键字调用父类方法时，实际上依旧是使用子类的对象获取锁。因此，若不是可重入锁，则会有死锁的情况发生。

实现原理

实现原理实现是通过为每个锁关联一个请求计数器和一个占有它的线程。当计数为0时，认为锁是未被占有的；线程请求一个未被占有的锁时，JVM将记录锁的占有者，并且将请求计数器置为1 。

如果同一个线程再次请求这个锁，计数器将递增；

每次占用线程退出同步块，计数器值将递减。直到计数器为0,锁被释放。

自旋锁(spinLock)

指尝试获取锁的线程不会立即阻塞，而是采用循环的方式去尝试获取锁。

例如Unsafe类中的getAndAddInt方法，使用的就是自旋的方式

当while条件不成立时，线程就换不断进行自旋来获取新的变量，并判断是否符合条件。

同时，自旋锁无法满足等待时间最长的线程优先获取锁，属于非公平锁。

优点在于自旋锁不会让线程状态发生切换，一直处于用户态，即线程一直都是active的，不会使线程进入阻塞状态，减少了不必要的上下文切换，因此执行速度快。

缺点在于如果线程占用锁的时间很长，那么会导致其他线程自旋而耗尽CPU。

同时，自旋锁概念本身并不支持可重入和公平性。但是，基于自旋锁可以额外实现此类功能。

Synchronized与ReentrantLock区别

  1. Synchronized属于JVM层面的关键字，ReentrantLock是JDK5开发的API接口
  2. Synchronized使用隐式锁，ReentrantLock需要手动加锁和释放锁
  3. Synchronized属于非公平锁，ReentrantLock默认非公平锁，但也可以设置为公平锁。
  4. Synchronized不可中断，除非抛出异常或运行完成。ReentrantLock可中断。通过设置tryLock(long timeout,TimeUnit unit)或者将lockInterruptibl()放入代码块中，再调用interrupt()方法可中断
  5. Synchronized不可绑定多个条件，ReentrantLock用来实现分组唤醒需要唤醒的线程们，可以精确唤醒，而不是像synchronized要么随机唤醒一个线程要么唤醒全部线程

CountDownLatch

作为一个同步工具类，用来协调多个线程之间的同步。能够使一个线程在等待另外一些线程完成各自工作之后，再继续执行。

使用一个计数器进行实现。计数器初始值为线程的数量。每当一个线程完成自己的任务之后，计数器的值就会减1。当计数器的值为0时，表示所有的线程都已经完成一些任务，然后在CountDownLatch上等待的线程就可以恢复执行接下来的任务。

值得注意的是，CountDownLatch是一次性的，计算器的值只能在构造方法中初始化一次，之后没有任何机制再次对其设置值，当CountDownLatch使用完毕后，不能被再次使用。

使用

CountDownLatch主要使用以下方法：

countDown()

他会递减锁存器的计数，如果计数达到0，则释放所有等待的线程。如果大于0，将计数减少

await()

让当前线程在锁存器的计数为0之前一直等待，除非线程被中断或超出了指定的等待时间或是计数为0，直接返回true。

CyclicBarrier

一组线程相互等待，当所有线程都到达某个屏障点后再进行后续的操作。

与CountDownLatch相似，但是区别在于：

  1. CyclicBarrier的计数器由自己控制，CountDownLatch由使用者来控制，在CyclicBarrier中线程调用await方法不仅会将自己阻塞还会将计数器减1，而CountDownLatch中线程调用await方法只是将自己阻塞而不会减少计数器的值。
  2. CountDownLatch只能拦截一轮，而CyclicBarrier可以实现循环拦截。

Semaphore

用来控制同时访问特定资源的线程数量，通过协调各个线程，以保证合理的使用资源。

当Semaphore进行初始化时，会创建一个非公平的锁的同步阻塞队列。将初始令牌数量赋给同步队列的state状态，其值就代表当前锁剩余的令牌数量。

当前线程会尝试获取令牌，其过程是使用原子操作去修改同步队列的state，获取一个令牌则修改为state=state-1。

当令牌数量不足时，会创建一个Node节点加入阻塞队列，挂起当前线程。

当调用release()方法时，线程会尝试释放一个令牌(state=state+1)。成功后，会唤醒同步队列中的一个线程。

被唤醒的节点会尝试修改state=state-1的操作，如果修改结果大于0获取成功，否则重新进入阻塞队列，挂起线程。

阻塞队列

阻塞队列是线程安全的。主要用于生产者消费者场景。

当生产者不断添加元素到队列中，消费者不断从队列中取出元素时。阻塞队列的作用主要是：

  • 在队列满的时候阻塞生产者添加元素
  • 在队列空的时候阻塞消费者取出元素

阻塞队列的四种处理方式：

抛出异常 返回特殊值 一直阻塞 超时退出
插入方法 add(o) offer(o) put(o) offer(o, timeout, timeunit)
移除方法 remove(o) poll() take(o) poll(o, timeout, timeunit)
检查方法 element() peek() — —

抛出异常

插入数据时，如果阻塞队列满，那么抛出异常IllegalStateException，否则插入成功返回true。

删除数据时，如果队列中有此数据，删除成功返回true，否则返回false。

检查队列时，如果队列为空，那么抛出异常NoSuchElementException。如果队列不为空，查询返回队列头部的数据，但是不移除数据

返回特殊值

插入数据时，如果阻塞队列没满，那么插入成功返回true，否则返回false。

移除数据时，如果队列不为空，查询、移除并返回队列头部元素。如果队列为空，那么返回null。

检查队列时，如果队列为空，返回null，这点不同于poll。如果队列不为空，查询返回队列头部的数据，但是不移除数据

一直阻塞

插入数据时，如果队列已满，那么阻塞等待队列可用，等待期间如果被中断，那么抛出InterruptedException。

删除数据时，查询、删除并返回队列头部元素，如果队列为空，那么阻塞等待队列可用，等待期间如果被中断，那么抛出InterruptedException。

超时退出

插入数据时，如果队列已满，那么阻塞指定时间等待队列可用，等待期间如果被中断，那么抛出InterruptedException。如果插入成功，那么返回true，如果在达到指定时间后仍然队列不可用，那么返回false。

删除数据时，查询、删除并返回队列头部元素，如果队列为空，那么阻塞指定时间等待队列可用，等待期间如果被中断，那么抛出InterruptedException。如果删除成功，那么返回队列头部元素，如果在达到指定时间后仍然队列不可用，那么返回null。

SynchronousQueue

也是一个队列，但特殊之处在于内部没有容器，当生产者生产一个元素后，若无线程消费。则进入阻塞状态，等待消费。

其内部没有使用AQS实现并发操作，而是使用CAS原理实现线程的安全访问。

线程池

在一个应用程序中，需要多次使用到线程，现成的创建和销毁会消耗内存。因此，通过线程池，可以减少内存的消耗。

线程池为一种管理线程的概念，通过设置线程的数量，存活时间等方法，来减少线程创建和销毁的次数，以此来减少内存消耗，更方便的管理线程。

执行流程

核心线程数：该线程即使没被使用也不会回收。

等待队列：遵循FIFO原则，当核心线程数满时，会将任务放入等待队列中排队执行。

线程池最大线程数：线程池可以存在的线程的最大数量。(核心线程+新创建的线程)

拒绝策略：分为四种

  1. AbortPolicy:不执行新任务，直接抛出异常，提示线程池已满
  2. DisCardPolicy:不执行新任务，也不抛出异常
  3. DisCardOldSetPolicy:将消息队列中的第一个任务替换为当前新进来的任务执行
  4. CallerRunsPolicy:直接调用execute来执行当前任务

创建方式

四种常用线程池：

CachedThreadPool:可缓存的线程池，该线程池中没有核心线程，非核心线程的数量为Integer.max_value，就是无限大，当有需要时创建线程来执行任务，没有需要时回收线程，适用于耗时少，任务量大的情况。

SecudleThreadPool:周期性执行任务的线程池，按照某种特定的计划执行线程中的任务，有核心线程，但也有非核心线程，非核心线程的大小也为无限大。适用于执行周期性的任务。

SingleThreadPool:只有一条线程来执行任务，适用于有顺序的任务的应用场景。

FixedThreadPool:定长的线程池，有核心线程，核心线程的即为最大的线程数量，没有非核心线程

核心线程数

线程数的设置的最主要的目的是为了充分并合理地使用 CPU 和内存等资源，从而最大限度地提高程序的性能。

使用 Runtime.getRuntime().availableProcessor() 方法来获取线程数。

而后需要判断任务类型

CPU密集型任务

像加解密，压缩、计算等一系列需要大量耗费 CPU 资源的任务，大部分场景下都是纯 CPU 计算。

对于 CPU 密集型的计算场景，理论上线程的数量 = CPU 核数就是最合适的，不过通常把线程的数量设置为CPU 核数 +1，会实现最优的利用率。

即使当密集型的线程由于偶尔的内存页失效或其他原因导致阻塞时，这个额外的线程也能确保 CPU 的时钟周期不会被浪费，从而保证 CPU 的利用率。

IO密集型任务

IO 密集型任务最大线程数一般会大于 CPU 核心数很多倍，因为 IO 读写速度相比于 CPU 的速度而言是比较慢的，如果我们设置过少的线程数，就可能导致 CPU 资源的浪费。

线程数 = CPU 核心数 * (1 + IO 耗时/ CPU 耗时)

或者

线程数 = CPU 核心数 / (1 - 阻塞系数)

总结

通过对线程数设置的探究，我们可以得知线程数的设置首先和 CPU 核心数有莫大关联，除此之外，我们需要根据任务类型的不同选择对应的策略，

线程的平均工作时间所占比例越高，就需要越少的线程；

线程的平均等待时间所占比例越高，就需要越多的线程；

针对不同的程序，进行对应的实际测试就可以得到最合适的选择。
